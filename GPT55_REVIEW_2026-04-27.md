### 1. A5 watchdog — race conditions, atomicity, lifecycle

Open `nuke_plugin/_watchdog.py`, `nuke_plugin/addon.py` (`_dispatch` hook), `src/nuke_mcp/connection.py` (`_consume_crash_marker`, `_consume_pending_warning`), `tests/test_watchdog.py`.

- The watchdog counter is global module state. What happens if two daemon threads dispatch concurrently inside the addon? Is there a lock?
- Atomic write: temp + `os.replace`. On Windows, does this raise if the destination is open by another process (e.g. a sibling MCP client reading the marker)? What's the recovery?
- `mtime within 1h` window: what if the system clock jumps backward (DST, NTP correction)? Does the marker stale-out incorrectly?
- The `try: from . import _watchdog / except ImportError` fallback in `addon.py`: if both branches succeed at different times in the same process, do they share state? Specifically: is the fallback's `spec_from_file_location` load a SECOND module instance with its own counter?

- Test isolation via `NUKE_MCP_MARKER_DIR`: does the test fixture clean up if pytest is interrupted (Ctrl-C)?

### 2. B5 Pydantic — schema correctness, wire compat, performance

Open `src/nuke_mcp/models/`, `src/nuke_mcp/tools/{read,render,knobs}.py`, `tests/test_models.py`.

- `NodeInfo`/`NodeSummary` use `class_: str = Field(alias="class")`. Does `model_dump(by_alias=True)` correctly emit `"class"` and NOT `"class_"`? Verify on a roundtrip.
- `KnobValue` typed as `int | float | str | bool | list[Any] | None`. Pydantic v2 unions resolve in declaration order — does `bool` get coerced to `int` (since `True == 1`)? If yes, that's a real bug; check `Field(strict=True)` or `Annotated[..., Field(...)]` discriminated.
- `model_validator(mode="before")` for `ScriptInfo` folds `first_frame`/`last_frame` into `frame_range`. Does this fire on `model_validate(dict)` AND on `dict()` round-trip? What if the addon ever sends BOTH `first_frame` AND `frame_range`?
- `extra="allow"` is permissive. Does it silently swallow a typo like `xpos` (when the field is `x` after alias)? Add an audit pass that warns on unexpected extras.
- `exclude_unset=True` on `model_dump`: this means ANY field not explicitly set by the addon is dropped from the wire. If the addon sometimes omits `errors` (instead of sending `errors=[]`), does the client see a missing key and crash? Walk every consumer.
- `tools/read.py` plumbing: the `read_comp` summary path returns `ScriptInfo`. The non-summary path returns the full nested dict. Are the two paths consistent? Does a client switching between them hit divergent schemas?
- Performance: every tool boundary now does `Pydantic.model_validate(dict) → model_dump(...)`. Pydantic v2 is fast, but on `read_node_detail` over a 5000-node script, the per-node parse+dump is non-trivial. Confirm with a microbenchmark or `cProfile` snapshot — flag if >5% of total request time.
- The `try: ... except Exception: return original` fallback in plumbing. This SILENCES validation errors. Add a warn-once log path (or bump to a structured warning) so a malformed addon payload doesn't go unnoticed in production.
