## Your job — twelve review angles

For each angle: find issues, edit, write regression test, run pytest, commit.

### 1. Pydantic v2 schema correctness across the new surface

`src/nuke_mcp/models/` was wired into 5 tools at B5 with `extra="allow"` and `model_dump(by_alias=True, exclude_none=True, exclude_unset=True)`. Now there are 96 tools, many of which return `NodeRef = {name, type, x, y, inputs}` (C1, C5, C6, C8). None of the C-phase tools wrap their dict in a Pydantic model.

- Should `NodeRef` be a Pydantic model? Pros: type safety, schema export. Cons: per-call overhead × N nodes per macro.
- Audit: where do tools return inconsistent shapes? E.g. `setup_flip_blood_comp_ss` returns NodeRef of the wrapper Group, but `audit_comp_for_acescct_consistency_ss` returns `{findings, sources}`. Document the divergences and propose a union.
- KnobValue discriminated union: is `bool` getting coerced to `int` via Pydantic v2 union resolution order? `True == 1` is the trap.
- `exclude_unset=True` audit: any field a downstream consumer relies on that the addon sometimes omits?

### 2. The Task primitive contract — cancellation, persistence, reconnect

B2 ships a state machine (working/input_required/completed/failed/cancelled), disk persistence at `~/.nuke_mcp/tasks/<id>.json`, demuxer, per-socket write lock, cooperative cancel, sweep on reconnect.

C4 SmartVector + STMap, C7 train_copycat + dehaze + install_cattery_model — six more Task-wrapped tools landed AFTER B2 was specified. Confirm:

- All six follow B2's listener self-unregister-on-terminal pattern (mirror of `tools/render.py:_on_progress`).
- Cancellation dispatch table covers the new `cancel_copycat` + `cancel_install` paths.
- Cooperative cancel between frames in the addon worker (no `nuke.cancel()` for non-render tasks — verify).
- `sweep_stale_working` on reconnect catches all six task kinds, not just `render`.
- Task progress fields are correctly allow-listed at the listener (no leaked private fields).

### 3. C8 Salt Spill macros — composition correctness

10 macros compose C2-C7 + C9 primitives via direct addon-handler calls. Spot-check that the inner handler is invoked exactly once per outer call (idempotency), and that the SubGraphRef + Backdrop wrapper actually wraps every node the inner handler created (not just the entry point).

- `setup_flip_blood_comp_ss` wraps `setup_flip_blood_comp` (C6) which wraps DeepRecolor + DeepHoldout2 + DeepMerge + ZDefocus. Does the C8 wrapper Group actually CONTAIN those inner nodes, or just reference the C6 sub-handler's return?
- `audit_comp_for_acescct_consistency_ss` — does it preserve the `severity` field on findings from each source, or homogenise?
- The new `source: str` field on findings (C8 deviation) — is it documented in the `findings` schema anywhere?

### 4. Profile pagination correctness with 10 profiles

`server.py:register_tools` registers all tool modules then `mcp.disable(names=...)` for non-active profiles. `load_profile(name)` runtime-enables.

- 10 profiles × 96 tools — is the `to_disable` set computed correctly, especially for tools that appear in multiple profile catalogues?
- `tasks_*` and `list_profiles/load_profile/unload_profile` are in `core` — are they SAFE to call when other profiles are unloaded? Specifically: if `tracking` profile is unloaded, can the `tasks_resume(id)` of a running C5 spaceship-track-patch find its handler?
- Empty profile case: load profile name not in `PROFILES` — graceful error or RuntimeError?

### 5. C10 prompt loader robustness

`prompts.py` reads `*.md` files from `src/nuke_mcp/prompts/` at registration. Custom regex `\{([A-Za-z_][A-Za-z0-9_]*)\}` for placeholders.

- What if a prompt body contains `{` in a code fence or JSON example with a non-identifier-shaped key (`{"foo-bar": ...}`)? The regex skips those, but verify.
- Hand-rolled YAML frontmatter scanner — what if a description has a colon in it (`description: "Foo: bar"`)?
- Wheel build: `pyproject.toml` doesn't include `package-data`, so the `.md` files won't ship in a wheel. Add it (or document as `# TODO(D)`).
- The `audit_acescct_consistency_guide` rename collision fix (post-merge): is the rename consistently applied across all tests, README references, and prompt-file frontmatter?

### 6. Cross-phase: Pydantic + watchdog + Tasks interaction

The watchdog's `_pending_warning` injection at `connection.py` happens BEFORE the Pydantic dump. If a Task response lifts the warning into the response dict, does Pydantic's `extra="allow"` preserve it? Trace one full path: render_async returns `{task_id, state, ack}`, listener emits progress → terminal → result merged into Task → MCP returns `{task_id}` initially.

- If reconnect fires mid-render, watchdog injects warning into the next response. That response is a `task_progress` notification. Notifications go through the demuxer, not the regular response path. So the warning is lost. Verify.

### 7. Full-codebase audit — drift, dead code, inconsistency

96 tools is a lot. Look for:

- **Dead code:** any addon handler with no MCP-side caller. Any tool registered but never tested (cross-check `tests/test_*.py` against the tool name set).
- **Param naming drift:** `node` vs `node_name` vs `name` vs `target_node`. Specifically the C5/C6/C8 macros — are all `name=` kwargs consistent?
- **Return shape divergence:** NodeRef vs `{status, ...}` vs Pydantic dump. Three patterns coexist; document and propose a union.
- **Validation drift:** every typed addon handler should validate inputs. Are there C2-C10 handlers that trust the wire blindly?
- **Annotation drift:** `EXPECTED_HINTS` snapshot covers all 96 tools — is every annotation correctly polarised? E.g. `audit_*` should be READ_ONLY; `setup_*` should be BENIGN_NEW.
- **Stale TODOs / FIXMEs:** find them, decide which are still real, fix or delete.

### 8. Connection layer with the new task surface

- Heartbeat thread: if 8 distinct Tasks are active concurrently (render + 2 SmartVector + 2 STMap + 1 train_copycat + 1 install_cattery_model + 1 dehaze), does the demuxer route all 8 progress streams correctly without dropping notifications?
- `_pending_warning` consumed by `send()` — but worker threads emit via the same socket. Does the warning get lifted into a worker progress notification by accident?
- Per-socket write lock: 8 worker threads × heartbeat × dispatch — is the lock granularity correct?

### 9. Test brittleness

- Snapshot test `test_no_unexpected_tools` will fire on every new tool addition. Fine, but check: does it clean up between runs (FastMCP component-already-exists from prior phases of the test session)?
- Task-progress mock injection in tests: does it deadlock or race when multiple listeners are registered simultaneously?
- `inspect.signature` pins on C1 + C5 + C6 + C8 macro tools — do they all use strict xfail / strict pin, or are some loose?

### 10. Schema / wire-format stability across v0.1.0 → v0.2.0

A client that was written against v0.1.0 — does v0.2.0 break them? Specifically:
- `read_node_detail` returns NodeInfo via Pydantic dump. v0.1.0 returned raw dict. Is the wire shape backward-compatible (extra keys are okay; missing keys break)?
- `render_frames` synchronous=True vs default async — v0.1.0 callers got finished result; v0.2.0 default returns task_id. Confirm `synchronous=True` is the documented compat path.
- `setup_aov_merge` was migrated channels.py → aov.py. Does the tool name change? If not, fine. If yes, breaking change — flag.

### 11. Performance + token shape

- 96 tools registered with FastMCP at boot, then 90% disabled by profile gate. What's cold-start time?
- `list_tools` / `prompts/list` — total bytes returned at boot. Should be <50KB.
- `read_comp` on a 2000-node script with B5 Pydantic round-trip. Confirm <100KB default.
- Pydantic model_dump on every C1+C5+C6+C8 NodeRef return — over-engineering or fine?

### 12. Hunt a real bug

Spend 10 minutes hunting a behavioural defect:
- `tools/salt_spill.py:setup_flip_blood_comp_ss` — when called with explicit `name=`, does the inner C6 macro also get `name=`? If not, the inner Group will have an auto-name and the C8 wrapper's `name=` mismatches its child.

- `tools/ml.py:install_cattery_model` — if the model is already cached locally, does the Task complete instantly with `state=completed` or does it spin a no-op worker?
- `connection.py` reconnect with 3 stale tasks AND a fresh crash marker — order of operations on the next response?
- C8 backdrop annotation: if a macro is called twice with the same `name=` (idempotency check), does the Backdrop also dedupe, or does it accumulate?
